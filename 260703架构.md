# 架构说明（v0.1.0 · 前后端分离版）

> 版本日期：2026-07-03
> 本文档描述 AILabeling-main 当前版本的整体架构，覆盖前端、独立后端、数据层、关键机制与部署方式。

---

## 一、概述

本项目是一个面向油气/地球物理（古地理）数据的 **AI Ops 平台原型**，核心是「数据与样本管理」模块——对单井/地震/切片三类地质数据做导入、存储、导出、删除。

- **形态**：前后端彻底分离。前端纯静态（React + Vite 构建），后端独立 Express 服务。
- **技术栈**：React 18 + TypeScript + Vite 5（前端）；Express + Node 内置 `node:sqlite` + `worker_threads`（后端）。
- **零额外重型依赖**：不依赖 MySQL / Redis / better-sqlite3；数据库用 Node 24 内置的 `node:sqlite`，文件转换用 SheetJS，仅此而已。

## 二、整体架构

### 开发期

```
浏览器 :5173  ──▶  Vite Dev Server ──proxy /api──▶  Express 后端 :3000
 (前端页面)         (热更新 + HMR)                    │
                                                      ├─▶ node:sqlite (data01/metadata.db, WAL)
                                                      ├─▶ 文件存储 data01/{danjing,dizhen,qiepian}
                                                      └─▶ worker_threads (XML→xlsx 转换)
```

- `npm run dev:all` 用 concurrently 同时起前端（:5173）和后端（:3000）。
- 前端 `/api/*` 请求经 Vite `server.proxy` 转发到后端，前端代码用同源相对路径，零配置。

### 生产期

```
浏览器 ──▶ Nginx ──┬─ 托管 dist/ 静态资源（前端）
                   └─ 反代 /api + limit_req 限流 ──▶ Express :3000（tsx 直跑）
                                                          │
                                                          └─▶ SQLite + 文件 + worker
```

- `npm run build` 产出 `dist/`（纯静态）。
- `npm start`（`tsx server/index.ts`）跑后端；Nginx 托管前端 + 反代后端 + 限流。
- 应急/单机：`SERVE_STATIC=true npm start`，让 Express 自己托管 `dist/`，单端口访问。

| path | 页面 | 数据来源 |

## 四、后端架构（`server/`）

独立 Express 服务，端口默认 3000（`PORT` 可改）。

### 4.1 模块组成

| 文件 | 职责 |
|------|------|
| `index.ts` | Express app；`express.raw` 上传中间件 + 4 个 `/api` 路由 + 静态托管开关（`SERVE_STATIC`）+ SIGTERM/SIGINT 优雅关闭 + `EADDRINUSE` 错误处理 |
| `db.ts` | 打开 `node:sqlite`；**WAL/PRAGMA 调优**；建表；`migrateDanjingIfNeeded`（一次性迁移）；`tx`（事务）；`listDanjing` / `importDanjing` / `deleteDanjing` / `getDanjingFile` / `closeDb` |
| `convert.ts` | `runConvert(buf)`：每次请求 new 一个 worker 做 XML→xlsx，**三道降级**到同步 `convertCore` |
| `convert.worker.ts` | worker 线程入口；监听 `parentPort`，调 `convertCore` 回传（用 `--import tsx` 加载 .ts） |
| `convertCore.ts` | XML→xlsx 核心逻辑（SheetJS）；worker 与降级路径共用 |
| `queue.ts` | 写入**内存串行队列** `enqueue(task)` + `flush()`（优雅关闭用） |
| `util.ts` | `formatBytes` / `sanitize` / `parseDatasetXml` / `listFiles`（文件夹扫描） |
| `types.ts` | `FileMeta` / `FolderKey` / `FOLDERS` 配置 / `isFolderKey` |

### 4.2 API（4 条路由，前端零改动沿用）

| 操作 | 方法 & 路径 | 说明 |
|------|-------------|------|
| 列表 | `GET /api/datasets?key=danjing\|dizhen\|qiepian` | 单井查库；地震/切片扫文件夹 |
| 下载 | `GET /api/datasets/:key/file?id=\|&name=` | 单井按 id 查 `stored_file`；其他按文件名；`res.download` 流式 |
| 导入 | `POST /api/datasets/:key?filename=`（body=原始字节） | 单井：xml→worker 转 xlsx→入库；其他：原样落盘 |
| 删除 | `DELETE /api/datasets/:key?id=\|&name=` | 单井删库记录 + 文件；其他删文件 |

- 上传用 `express.raw({type:'*/*', limit:'200mb'})`，前端把 `File` 作为原始字节 body 发送（非 multipart）。
- 路径安全：所有按文件名操作经 `path.basename` 拦截 `../`；id 用 `/^\d++$/` 校验。

### 4.3 导入请求处理流程（单井，最复杂路径）

```
POST /api/datasets/danjing?filename=xxx.xml  (body = 文件字节)
  │
  ├─ express.raw 收到 Buffer
  ├─ 已是 .xlsx？ ─ 是 ─▶ 直接用 buf
  │                ─ 否 ─▶ runConvert(buf)
  │                          ├─ new Worker(convert.worker.ts, execArgv:['--import','tsx'])
  │                          ├─ worker: convertCore(xml→xlsx)，postMessage 回传
  │                          └─ worker 失败（spawn错/运行错/30s超时）→ 三道降级同步 convertCore
  ├─ enqueue(() => importDanjing(data, name))   ← 写入串行队列
  │     └─ 事务：INSERT → 拿 lastInsertRowid → 写 <id>.xlsx → UPDATE stored_file
  └─ 返回 { ok, id, savedAs:'<id>.xlsx', converted:true|false }
```

## 五、数据层

### 5.1 三类数据的存储方式

| 类型 | key | 存储 | 列表来源 | 转换 |
|------|-----|------|----------|------|
| 单井数据 | `danjing` | SQLite 元数据 + `<id>.xlsx` 文件 | 查库（毫秒级） | xml→xlsx（worker） |
| 地震数据 | `dizhen` | 文件夹原格式 | 扫文件夹 | 不转 |
| 切片数据 | `qiepian` | 文件夹原格式 | 扫文件夹 | 不转 |

### 5.2 单井数据模型（SQLite，`data01/metadata.db`）

```sql
CREATE TABLE danjing_dataset (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,  -- 主键，自增
  name        TEXT NOT NULL,        -- 友好名（界面显示用）
  format      TEXT,                 -- 'xlsx'
  size_bytes  INTEGER,
  created_at  TEXT NOT NULL,        -- YYYY-MM-DD
  status      TEXT NOT NULL DEFAULT '原始',
  stored_file TEXT NOT NULL         -- 磁盘文件名，固定 '<id>.xlsx'
);
```

**核心约定：文件名 = `<id>.xlsx`**。id 是唯一锚点，所有"找到某一条"都靠 `WHERE id = ?`；找到 id 即找到文件，无需额外映射表。详见 `zsgc.md`。

### 5.3 存储目录

```
data01/
├── metadata.db            # SQLite（单井元数据）
├── metadata.db-wal        # WAL 辅助（勿手动删）
├── metadata.db-shm        # WAL 辅助
├── danjing/<id>.xlsx      # 单井数据文件
├── dizhen/                # 地震（原格式，如 .xml）
└── qiepian/               # 切片（原格式）
```

## 六、关键机制（面向 300–400 人并发的加固）

### 6.1 SQLite 调优（`db.ts` 启动时执行）

```sql
PRAGMA journal_mode = WAL;     -- 读写不互锁
PRAGMA synchronous = NORMAL;   -- WAL 下安全且快
PRAGMA cache_size = -64000;    -- 64MB 页缓存
PRAGMA temp_store = MEMORY;
```

### 6.2 CPU 密集任务进 worker 线程

XML→xlsx（SheetJS）是 CPU 密集，若在主路由同步做，多人同时导入会把 Express 主线程占满、卡死所有请求。故挪到 `worker_threads`：

- 每次请求 new 一个 worker，跑完 terminate（无状态、无并发竞态）。
- 子线程加载 `.ts` 需 `execArgv: ['--import', 'tsx']`（Node 原生 worker 不认 .ts）。
- **三道降级**（保证导入永不中断）：① spawn 抛错 ② worker 运行错 ③ 30s 超时 → 均退回主线程同步 `convertCore`。
- 降级同步若也失败（数据损坏），则 reject 让路由返回 500，**绝不抛未捕获异常崩进程**。

### 6.3 写入内存串行队列（`queue.ts`）

SQLite 是单写者模型，并发写会撞 `SQLITE_BUSY`。所有写操作（导入/删除）经 `enqueue(task)` 串行化：

- 接口仍同步等结果返回（前端零改动）。
- 队列失败打 `[queue] task failed` 日志，不卡后续任务。
- 为将来"立即返回 + 后台处理"留好入口：把 `await enqueue(...)` 改成入队返回 taskId 即可。

### 6.4 优雅关闭（`index.ts`）

监听 `SIGTERM`/`SIGINT`：停 HTTP → `await flush()`（等队列排空）→ `closeDb()` → `process.exit(0)`。避免重启时强杀 worker、丢队列任务、SQLite 残锁。

> ⚠ Windows 信号支持有限（SIGTERM 不一定触发 handler，Git Bash 的 `kill` 不识别原生 PID）；生产请部署在 **Linux + PM2**，`pm2 reload` 发 SIGINT 优雅重启。

### 6.5 端口冲突处理

`app.listen` 的 `error` 事件捕获 `EADDRINUSE`，输出清晰提示并 `exit(1)`，而非打印崩溃栈。

## 七、开发与部署

### 7.1 npm 脚本

| 命令 | 作用 |
|------|------|
| `npm run dev` | 前端开发服务器（:5173，HMR） |
| `npm run server` | 后端开发服务器（:3000，tsx watch 热重载） |
| `npm run dev:all` | 一键同时起前后端（concurrently） |
| `npm run build` | 前端 tsc 类型检查 + vite 打包 → `dist/` |
| `npm start` | 生产：启动后端（tsx server/index.ts） |
| `npm run typecheck:server` | 仅后端 `server/` 类型检查 |

### 7.2 类型检查拆分

- `tsconfig.json`（前端）：`exclude: ["server"]`，避免前端 tsc 误检后端 node 代码。
- `tsconfig.server.json`（后端）：`target ES2022 / moduleResolution bundler / esModuleInterop / noEmit`。
- 选 **tsx 生产直跑** 而非编译：dev/生产代码与 worker 加载方式完全一致，省去 esbuild 处理 worker 入口的坑。

### 7.3 生产部署要点

- 推荐 Nginx 托管 `dist/` + 反代 `/api` + `limit_req` 限流（配置见 `docs/deploy.md`）。
- 硬件：4 核 / 4GB / **SSD**（SSD 对并发写入体验影响最大）。
- SQLite 备份时连同 `.db-wal` 一起，或先 `PRAGMA wal_checkpoint(TRUNCATE)`。

## 九、已知问题与后续演进

### 已知问题（不影响运行）
- **CDN/importmap 与 package.json 版本不一致**：importmap 指 React 19 / Router 7 / Recharts 3，`package.json` 声明 18 / 6 / 2。运行时以 CDN 为准，建议后续统一（要么去掉 importmap 走 npm 构建，要么版本对齐）。
- **生产依赖外网 CDN**：断网/内网部署 UI 会崩。生产建议改为本地构建链（PostCSS + npm 包）。
- **`node:sqlite` 为实验性 API**：启动有 `ExperimentalWarning`，无害。

### 后续演进路径（按需）
- **地震/切片入库**：当前扫文件夹，可照单井模式加表，统一 CRUD。
- **「修改」功能**：当前只有增/查/删，可加 `PATCH /api/datasets/:key?id=` 改名/改状态。
- **写入异步化**：队列已就绪，将来要"提交成功、后台处理中"体验时，入队返回 taskId + 加任务状态查询接口。
- **持久化队列**：若并发写入 >200 TPS，可引入 BullMQ + Redis（会引入 Redis 依赖）。
- **认证鉴权**：当前无，多用户协作前需补。
