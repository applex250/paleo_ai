# 模型上传与统一评估：方案讨论与结论

> 面向：平台技术团队评审决策
> 状态：待评审 → 评审通过后进入排期
> 结论速览：**采用方案二**。以「AI 辅助上传（即方案一）」为默认入口、以「手动对齐」为兜底；两条路径最终产出**同一种标准适配器**，由平台统一的评估引擎计算指标。
> 关键约束：AI 只能**生成适配器**，**绝不改写模型推理本体**。

---

## 一、背景与问题

平台要让各团队把训练好的模型上传到统一服务器上运行，并产出**可横向对比的评估指标**（对应平台现有指标：Accuracy / Precision / Recall / F1 / IoU，以及业务场景如地震相、单井相、岩心分类）。

现实难点：

- 各团队的代码风格、依赖、框架（PyTorch / ONNX / HuggingFace / 自研）、**输入输出格式、验证(val)与评估(eval)写法各不相同**。
- 如果任由上传方自带评估代码，则"指标"不可比：同一个 F1 可能用了不同的阈值、不同的 averaging、不同的 ground-truth 对齐方式。
- 因此核心问题不是"怎么把代码跑起来"，而是 **"怎么让评估指标的算法本身被平台独占、统一、不被上传方篡改"**。

---

## 二、核心目标与设计原则

| # | 原则 | 说明 |
|---|------|------|
| P1 | **指标统一在"契约层"完成** | 平台定义一套标准 I/O 与评估契约；任何模型无论怎么上传，最终都必须满足该契约。统一性来自契约，不来自"改写源码"。 |
| P2 | **责任切分** | 上传方只负责"输入输出对齐"；**验证集与指标实现由平台独占**。这是"统一计算"的根本保证。 |
| P3 | **模型本体不可变** | 评估的是"原模型"，不是"被改写过的模型"。任何自动化处理只能加包装，不能动推理逻辑。 |
| P4 | **可追溯、可调试** | 生成的适配器是一份可见、可 diff、可人工修正的源码产物，不是黑盒运行时改写。 |
| P5 | **覆盖面优先于一致性洁癖** | 必须为复杂/非标准模型保留人工兜底，避免"跑不起来"被卡死。 |

> P1 + P2 直接回答了"怎么做到指标统一计算"：**把"对齐 I/O"和"计算指标"在架构上彻底分开**，契约就是两者的接缝。

---

## 三、两个方案概述

```
方案一（纯 AI 改写）：所有人 → 大模型按平台规范重写代码 → 统一 val + 统一评估

方案二（双路径）：上传人二选一
   ├─ 路径 A：AI 辅助上传（即方案一，作为默认推荐）
   └─ 路径 B：手动对齐（上传人自行实现/选择标准 val 方案）
```

**重要澄清**：方案二并非与方案一并列，而是**包含方案一**。真正的选择是——

> 是否保留一条**人工兜底路径**？

- 选"否"= 方案一：全员走 AI。
- 选"是"= 方案二：AI 为默认，人工为兜底。

---

## 四、方案对比

| 维度 | 方案一（全员 AI 改写） | 方案二（AI 默认 + 手动兜底） |
|------|------------------------|------------------------------|
| 指标统一性 | 统一 | **统一**（两条路径收敛到同一契约 + 同一评估引擎） |
| 评估可信度 | **有风险**：若 AI 动了推理本体，结果不可信 | 高：模型本体不变，AI 只生成适配器 |
| 复杂/非标模型覆盖 | 差：AI 改写失败即卡死 | 好：可退回人工对齐 |
| 上传方心智负担 | 低 | 低（默认走 AI，与方案一一致） |
| 平台代码路径数 | 1 | **仍是 1**（产物同为 `ModelAdapter`，评估引擎不分支） |
| 可调试/可追溯 | 中（取决于是否保留 diff） | 好（适配器为可见源码） |
| 工程实现复杂度 | 低 | 中（多一个适配器 SDK + 模板） |
| 适合的团队 | 内部小范围、模型高度同质 | 多团队、模型异构、需对外可信 |

---

## 五、关键判断（为什么不是方案一）

1. **"改写代码"会破坏评估的公信力。** 评估平台的全部价值在于指标可比、可信。一旦允许 LLM 改写模型推理代码，就存在"你评的到底是哪个模型"的争议——这在科研/横向对比场景是不可接受的。

2. **AI 改写在非标模型上会失败。** 自定义算子、C++ 扩展、复杂的前后处理、多输入分支……这些场景下 LLM 大概率改不动或改错。全员走 AI 意味着这类模型直接无法上传，扼杀覆盖面。

3. **真正需要统一的是"接口"，不是"源码"。** 只要让每个模型暴露同一个 `predict` 契约，评估引擎就能统一计算指标。达成这一点，**用"生成适配器"就足够**，无需触碰模型本体。

4. **只要两条路径产出同一种适配器，方案二就不会增加评估侧的复杂度。** 这是把方案二的"双路径"成本压缩到最小的关键设计（见下文 §6.3）。

> 因此方案一在"全员统一"上看似最干净，但它把统一性建立在"改写别人的代码"这一不可靠、不可信的基础上，且牺牲了覆盖面。方案二保留了方案一的全部便利（默认走 AI），同时补上了正确性与覆盖面两个短板。

## 六、推荐方案：方案二的技术设计

### 6.1 总体架构（一图）

```
┌──────────────────────── 上传方拥有的东西（不可信区） ────────────────────────┐
│  原始模型代码 / 权重 / 原生 preprocess·infer·postprocess                       │
└───────────────┬──────────────────────────────────────────┬───────────────────┘
                │ 路径 A：AI 辅助（默认）                      │ 路径 B：手动对齐
                │ LLM 阅读仓库，生成 adapter.py                │ 上传方按 SDK 自行实现
                │ （仅包装，不改模型本体）                      │ 或选框架模板填 I/O 映射
                ▼                                              ▼
        ┌─────────────────────────────────────────────────────────┐
        │           标准适配器 ModelAdapter（唯一产物）            │  ← 两路径同产物
        │  predict_batch(StandardInput[]) -> StandardOutput[]      │
        └───────────────────────────────┬─────────────────────────┘
                                         │ 平台独占区（可信）
        ┌────────────────────────────────▼────────────────────────┐
        │  评估引擎（平台统一实现，上传方不可篡改）                  │
        │  · 标准验证集 ValDataset（按任务类型）                    │
        │  · 指标实现 Metrics（版本固定，如 IoU@v1、F1-macro@v1）   │
        │  · 对账与异常剔除                                          │
        └───────────────────────────────────┬─────────────────────┘
                                            ▼
                                  统一评估报告 EvalReport
```

**核心机制**：上传方与平台之间的唯一接缝是 `ModelAdapter`。平台只调用 `predict_batch`，只用自己的验证集和指标实现，因此 **"统一计算"在架构上被强制保证**，与上传方走哪条路径无关。

### 6.2 接口契约（关键细节）

平台定义三种按**任务类型**约定的标准类型（伪代码，最终用 Python Protocol + JSON Schema 落地）：

```python
# —— 标准输入/输出，按任务类型区分 ——
class StandardInput(TypedDict):
    sample_id: str
    payload: dict        # 分割: {"image": ndarray/uri}; 分类: {"features": ...}; 曲线: {"curves": {...}}

class StandardOutput(TypedDict):
    sample_id: str
    prediction: dict     # 分割: {"mask": ..., "classes": [...]}
                        # 分类: {"label": str, "probs": [...]}
    meta: dict           # 推理耗时、所用阈值等，便于排查

# —— 唯一的模型接缝 ——
class ModelAdapter(Protocol):
    model_version: str
    task_type: TaskType           # SEGMENTATION | CLASSIFICATION | SEQUENCE ...

    def load(self, artifact_dir: str) -> None: ...
    def predict_batch(self, items: list[StandardInput]) -> list[StandardOutput]: ...
```

```python
# —— 评估侧：平台独占，上传方无权提供实现 ——
class ValDataset(Protocol):
    task_type: TaskType
    def __iter__(self) -> Iterator[tuple[StandardInput, GroundTruth]]: ...

def evaluate(adapter: ModelAdapter, valset: ValDataset,
             metrics: list[MetricSpec]) -> EvalReport: ...   # 平台实现，封死
```

> 上传方**不能**提交自己的 `evaluate`。他只能从平台提供的、版本固定的指标清单里按任务类型**选用**（如分割必报 IoU、mDice；分类必报 Accuracy、F1-macro）。这正是"指标统一"的硬保证。

### 6.3 双路径同产物（控制方案二的复杂度）

- **路径 A（AI 辅助，默认）**：上传一个仓库或压缩包，LLM 读取入口、前后处理与依赖，**生成一份 `adapter.py`**，把原生调用包装成 `ModelAdapter`。**不修改任何模型文件**。生成结果以**可 review 的 diff / PR 形式**呈现，上传方可改、可回退。
- **路径 B（手动对齐，兜底）**：上传方在平台 SDK 里自行实现 `ModelAdapter`；平台为常见框架（HF Transformers、ONNX、原生 Torch、TensorRT）提供**模板**，上传方只需填"我的输入字段 → StandardInput、我的输出 → StandardOutput"的映射，并选用对应的标准 val 方案。

无论 A 还是 B，**产物都是同一个 `ModelAdapter`**，下游评估引擎完全一致。平台因此只维护一套评估链路，方案二"双路径"的维护成本被压缩到"多一个适配器 SDK + 几个框架模板"。

### 6.4 AI 辅助路径的硬约束（防止方案一的风险在方案二里复现）

1. **只生成适配器，不动推理本体**：AI 可读、可调用模型，但禁止修改 `model/*.py`、权重、自定义算子；生成物仅限 `adapter.py` + 一份依赖声明。
2. **生成物必须可运行自检**：在沙箱里用 1~2 个样本跑通 `predict_batch`，I/O 与契约不符即报错并提示，不让脏产物进入评估。
3. **可追溯**：保留 LLM 的输入(Prompt/仓库快照)与输出(adapter 源码)，便于事后排查"是模型问题还是适配器问题"。
4. **可选人工接管**：AI 产物默认进路径 B 的编辑器，上传方一键转人工微调——即 A、B 不是两个孤岛，A 的失败可平滑落到 B。

## 七、实施阶段（分期，最小闭环优先）

| 阶段 | 目标 | 产出 |
|------|------|------|
| **P0 契约与引擎** | 定义 `ModelAdapter`/`StandardInput`/`StandardOutput`/`evaluate` 契约；实现平台独占的评估引擎 + 至少 2 类任务(分割/分类)的标准 val 集与指标 | 即便只有路径 B，也能交付**可信、可对比**的评估 |
| **P1 路径 B（手动）** | 适配器 SDK + 框架模板(HF/ONNX/Torch)；上传/校验/沙箱运行流水线 | 真正可用的上传入口，覆盖所有模型 |
| **P2 路径 A（AI 辅助）** | 接入 LLM 生成 `adapter.py`；沙箱自检；可 review/diff；A→B 平滑降级 | 把 80% 简单模型的上传成本降到接近"传个包" |
| **P3 治理** | 指标版本化、模型版本库联动、评估报告导出、异常样本对账 | 长期可信运营 |

> 排期建议：P0+P1 即可解锁"统一评估"的核心价值（已能产出可比指标）；P2 是体验优化，可并行。这样即使 AI 路径遇到困难，平台目标不受阻塞。

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| AI 生成的适配器有隐性 bug（如把 mask 类别顺序搞反） | 指标失真 | 沙箱自检 + 少量"金标准"样本对账 + 适配器 diff 强制 review |
| 上传方在 `predict_batch` 里偷塞测试集信息 / 阈值作弊 | 指标虚高 | 验证集由平台掌握、对上传方不可见；记录 `meta` 中的阈值与耗时做异常检测 |
| 某任务类型没有标准 val 集 | 无法评估 | P0 先覆盖主流类型(分割/分类)；其余类型显式标"暂不可评"，不硬上 |
| 适配器契约随业务演进导致历史结果不可复现 | 横向对比失效 | 契约与指标实现都**版本化**，报告中固定记录所用版本 |
| AI 改写诱惑（团队图省事去动模型本体） | 破坏可信度 | §6.4 写进规范与代码门禁；CI 校验模型权重哈希一致 |

## 九、待团队决策的开放问题

1. 首批覆盖哪几种**任务类型**？（建议：图像分割、图像分类、曲线/序列标注——对齐现有地震相/单井相/岩心场景）
2. 标准验证集由**平台统一准备**，还是允许"上传方自带 + 平台校验"双轨？（影响 P0 工作量与公信力）
3. AI 辅助路径用**哪个模型/部署形态**（自建 vs API）？涉及代码与权重出域的风险评估。
4. 适配器运行**沙箱与算力配额**（CPU/GPU、显存上限、超时）如何定？
5. 是否需要支持**增量上传/版本迭代**并自动重评，与现有模型注册库打通？

---

## 附：一句话结论

> 采用 **方案二**：AI 辅助为默认上传入口、手动对齐为兜底，二者产出**同一个标准适配器**；**验证集与指标算法由平台独占、版本固定**。模型本体一行不改，统一性由"契约"而非"改写"来保证——这样既拿到方案一的便利，又保住评估的可信度与对复杂模型的覆盖面。
