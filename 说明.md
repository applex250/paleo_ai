# 本地预览说明

> 本文档已**实际验证**可跑通（Node v24 / npm 11，Vite v5.4.21，Express 后端 :3000，页面完整渲染）。
> 项目：React + TypeScript + Vite 前端 + 独立 Express 后端（`server/`，Node 内置 `node:sqlite`）。前端样式与部分依赖走 CDN。

---

## 一、环境要求

| 项 | 要求 | 说明 |
|----|------|------|
| Node.js | 建议 20 及以上（实测 v24，用到内置 `node:sqlite`） | 自带 npm |
| 网络 | **必须能访问外网** | 前端 React/Recharts/Tailwind 等经 CDN 加载（见 `index.html` importmap）。断网会白屏。 |

```bash
node -v
npm -v
```

## 二、快速开始（前后端分离）

首次运行先装依赖：

```bash
npm install
```

本项目是**前后端分离**：前端 Vite（:5173），后端 Express（:3000，含 SQLite + 文件存储）。开发期 Vite proxy 把 `/api` 转发到后端，前端零配置。

**方式 A（推荐，一键起前后端）**：
```bash
npm run dev:all
```
同时启动前端（:5173）和后端（:3000，热重载）。

**方式 B（分两个终端）**：
```bash
npm run server   # 终端1：后端（热重载）
npm run dev      # 终端2：前端
```

浏览器打开 **http://localhost:5173/**（前端地址；`/api` 自动转发到后端）。

**启动成功的标志**（看到这些日志说明就绪）：
- 前端：`VITE v5.4.21 ready ... ➜ Local: http://localhost:5173/`
- 后端：`[api] listening on :3000`（首次启动若 `data01/danjing/` 已有 xlsx，还会打印一行 `[db] 已把…登记进 SQLite 并改名为 <id>.xlsx`）
- `dev:all` 下两条日志带颜色前缀 `[web]` / `[api]`，便于区分前后端。

## 三、停止正在运行的服务

**前台运行**：对应终端按 `Ctrl + C`。

**后台残留 / 找不到终端**（典型表现：再次启动提示 5173 或 3000 被占用），按端口杀进程：

```bash
# 跨平台，自动按端口杀
npx kill-port 5173 3000
```
```bash
# Windows 手动定位（Git Bash，flag 用双斜杠 //F）
netstat -ano | grep :3000
taskkill //PID <PID> //F
```

> 后端用 tsx watch（`npm run server`）；改 `server/` 下代码会自动重启。
> ⚠ 若后端进程异常退出（如端口冲突未被捕获），tsx watch 不一定回收端口，会留下残留进程占着 3000（实测再次启动报 `EADDRINUSE`）。此时用上面的 `netstat`+`taskkill` 杀掉再重启。

## 四、生产构建与部署

```bash
npm run build     # 前端类型检查 + 打包 → dist/
npm start         # 启动后端（tsx 直跑 server/index.ts）
```

- 推荐：**Nginx 托管 dist/ 静态资源 + 反代 /api 到 Express:3000 + limit_req 限流**（配置见 `docs/deploy.md`）。
- 应急/单机：`SERVE_STATIC=true npm start`，让 Express 自己托管 dist/，单端口访问 http://localhost:3000/。
- 端口可用 `PORT` 环境变量改：`PORT=8080 npm start`。

## 六、常见问题

**1. 打开白屏 / CDN 加载失败**
网络无法访问外网 CDN（`aistudiocdn.com`、`cdn.tailwindcss.com`）。确保外网可达。

**2. 后端报 "端口 3000 已被占用"（EADDRINUSE）**
残留进程未关。用第三节方法杀 3000，或换端口 `PORT=3001 npm start`。

**3. 前端能开但「数据管理」页报错 / 列表空**
后端没起。开发期前端和后端都要跑（用 `npm run dev:all`，或分别 `npm run server` + `npm run dev`）。

**4. `npm install` 慢**
换国内镜像：`npm config set registry https://registry.npmmirror.com`。

## 七、数据存储与后端（独立 Express 服务）

「数据与样本管理」页（`/#/data`）由**独立后端服务**（`server/`，Express + node:sqlite）驱动：

- 存储目录：项目根 `data01/`，按类型三个子文件夹：`danjing/`（单井）、`dizhen/`（地震）、`qiepian/`（切片）。
- **单井数据（danjing）**：用 SQLite（`data01/metadata.db`）管理元数据，文件统一存为 `<id>.xlsx`；导入 `.xml` 经 **worker 线程**转 `.xlsx`，`.xlsx` 原样入库。地震/切片按文件夹扫描、原格式存储。
- 性能加固（面向并发）：SQLite 开 WAL + PRAGMA 调优；XML→xlsx 转换在 worker 线程做（不阻塞主线程，失败自动降级同步）；写入经内存串行队列（避免并发写 SQLITE_BUSY）。
- 接口：`/api/datasets/*`（增删改查 + 文件上传/下载），详见 `zsgc.md`。
- **怎么确认 XML→xlsx 转换在 worker 子线程跑**（而非降级到主线程同步）：导入一个 `.xml` 后，后端日志应出现 `[worker] started tid N`，且**没有** `[convert] ...，降级同步`。若看到降级日志，说明 worker 没起来——功能仍正常（自动退回同步转换），只是失去了"不阻塞主线程"的优势。

> 后端代码在 `server/`（`index.ts` 路由 / `db.ts` 数据库 / `convert.ts` worker 转换 / `queue.ts` 写入队列）。已与前端彻底分离：`npm run build` 产出纯静态 dist/，由 Nginx 或 Express 静态托管。

## 八、已知工程提示（不影响运行，供后续清理）

- `index.html` 的 importmap 指向 **React 19**（CDN），而 `package.json` 声明 **React 18**。本地 `npm run dev` 实测正常（浏览器按 importmap 用 CDN 版本），建议后续统一。
- Tailwind 目前走 CDN（`cdn.tailwindcss.com`），生产建议改为本地 PostCSS 构建链。
- `node:sqlite` 为 Node 实验性 API，启动会有 `ExperimentalWarning`，无害。

## 九、启动与功能测试

启动前后端后，用以下方法验证功能是否正常。

### 9.1 启动并确认就绪

```bash
npm run dev:all      # 一键起前端(:5173) + 后端(:3000)
```

看到这两条日志即说明就绪：
- 前端：`VITE v5.4.21 ready ... ➜ Local: http://localhost:5173/`
- 后端：`[api] listening on :3000`

（`dev:all` 下每行行首还会带 concurrently 的 `[web]` / `[api]` 颜色前缀，便于区分。）

### 9.2 浏览器端测试（最直观）

打开 http://localhost:5173/ → 左侧「数据与样本管理」：

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 列表 | 切换「单井/地震/切片数据」 | 各显示对应 `data01/` 文件夹的真实文件 |
| 导入 | 点「数据导入」选本地 `.xml`/`.xlsx` | 列表新增一条，顶部绿色「已导入」提示 |
| 导出 | 点某行「数据导出」 | 浏览器下载该文件（单井下载 `<友好名>.xlsx`） |
| 删除 | 点某行红色「删除」→ 确认 | 列表减少，`data01/` 对应文件消失 |

### 9.3 命令行测试（curl，直接验证后端接口）

后端在 :3000，可不经前端直接 curl 测：

```bash
# 1) 列表（单井查库；地震/切片扫文件夹）
curl -s "http://localhost:3000/api/datasets?key=danjing"

# 2) 导入 xml（单井会触发 worker 转 xlsx）→ 期望 {"ok":true,"converted":true,...}
curl -X POST "http://localhost:3000/api/datasets/danjing?filename=test.xml" \
  -H "Content-Type: application/octet-stream" \
  --data-binary @某个文件.xml

# 3) 导入 xlsx（不触发 worker）→ 期望 "converted":false
curl -X POST "http://localhost:3000/api/datasets/danjing?filename=test.xlsx" \
  --data-binary @某个文件.xlsx

# 4) 下载（按 id）
curl -o dl.xlsx "http://localhost:3000/api/datasets/danjing/file?id=<id>"

# 5) 删除（按 id）
curl -X DELETE "http://localhost:3000/api/datasets/danjing?id=<id>"
```

> 地震/切片把路径里的 `danjing` 换成 `dizhen`/`qiepian`，且用 `?name=<文件名>` 代替 `?id=`。

### 9.4 确认 worker 在子线程跑（性能加固验证）

导入 `.xml` 后看后端终端日志：
- ✅ 正常：出现 `[worker] started tid N`，且**没有** `[convert] ...，降级同步`
- ⚠ 降级：出现 `[convert] ...，降级同步`，说明 worker 没起来（功能仍正常，只是退回主线程同步转换，失去了"不阻塞主线程"的优势）

### 9.5 生产模式测试

```bash
npm run build                    # 构建前端 → dist/
SERVE_STATIC=true npm start      # 后端 + 同端口托管 dist/
```
访问 http://localhost:3000/ → 前端页面 + `/api` 全通（同源，无跨域）。

### 9.6 类型检查（无需启动服务）

```bash
npm run typecheck:server   # 仅后端 server/ 类型检查
npm run build              # 前端 tsc + vite build（顺带前端类型检查）
```
